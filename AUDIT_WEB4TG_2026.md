# WEB4TG — Стратегический аудит продукта
## CPO + Lead UX/UI + Senior Tech Architect | Весна 2026

---

## 1. EXECUTIVE SUMMARY

**WEB4TG** — это Telegram Mini App, позиционирующая себя как премиум-витрина и маркетплейс бизнес-решений для экосистемы Telegram. Приложение демонстрирует амбициозную техническую базу (React 19, Drizzle ORM, адаптивная производительность, предиктивный префетч), но имеет ряд критических проблем, мешающих выходу на мировой уровень.

### Текущая оценка (из 10):

| Категория | Оценка | Комментарий |
|---|---|---|
| **Продуктовая стратегия** | 5/10 | Размытый фокус: витрина, маркетплейс, геймификация, AI, конструктор — всё сразу |
| **UX/UI качество** | 7/10 | Визуально премиальный, но онбординг блокирует доступ ко всему контенту |
| **Техническая архитектура** | 6/10 | Продвинутые паттерны, но 119 зависимостей и 203 компонента — перегруженность |
| **Telegram-интеграция** | 7/10 | Глубокая интеграция, но не используется TON и новейшие API 2026 |
| **Масштабируемость** | 6/10 | Redis + PG, но A/B тесты в памяти, нет CDN для медиа |
| **Безопасность** | 8/10 | Многоуровневый rate limiting, CSRF, Helmet — сильная сторона |

**Общий балл: 6.5/10** — крепкий технический фундамент, но продукт пытается быть всем сразу, что размывает ценностное предложение.

## 2. КРИТИЧЕСКИЕ ПРОБЛЕМЫ (P0 — Требуют немедленного решения)

### 2.1 Онбординг блокирует весь продукт
**Проблема:** Полноэкранный оверлей онбординга (`OnboardingFlow`) перекрывает всё приложение. При каждом заходе (в режиме preview/dev) пользователь видит только "Welcome to WEB4TG" и не может взаимодействовать с продуктом, пока не пройдёт все 3 шага. Навигация по URL (`/projects`, `/ai-process`, `/profile`) тоже заблокирована — все страницы показывают один и тот же оверлей.

**Влияние:**
- 0% видимости продукта при первом контакте
- Невозможность deep-linking из Telegram (ссылки типа `t.me/bot?startapp=demo_id` бесполезны)
- Нарушение основного UX-принципа: "Show value before asking for commitment"

**Рекомендация:**
- Заменить блокирующий оверлей на неинвазивный "coach marks" тур, запускающийся после 2-3 секунд бездействия
- Или использовать встроенные подсказки (tooltips) при первом взаимодействии с каждой фичей
- При deep-link переходах — полностью отключить онбординг

### 2.2 CloudStorage как единственный источник истины
**Проблема:** Только что исправленный баг — `CloudStorage.getItem` вызывался без обработки ошибок и ронял всё приложение. Это симптом системной проблемы: SDK Telegram v6.0 не поддерживает CloudStorage (требуется >=6.9), но код полагался на `?.` вместо полноценной проверки версии.

**Влияние:** Полный краш приложения в средах с Telegram API < 6.9

**Рекомендация:**
- Создать единый адаптер `TelegramStorageAdapter` с автоматическим определением доступных API
- Паттерн: CloudStorage → localStorage → серверный fallback (через API)
- Применить к ВСЕМ местам использования (сейчас OnboardingFlow и OnboardingTutorial используют разные подходы)

### 2.3 119 production-зависимостей
**Проблема:** Приложение использует 119 npm-зависимостей в production. Для Telegram Mini App это критически много. Среди них:
- `@google-cloud/storage` — тяжёлая GCS SDK (нужна только на сервере)
- `three` + `@react-three/fiber` + `@react-three/drei` — 3D-движок для (предположительно) минимального использования
- `@splinetool/react-spline` + `@splinetool/runtime` — ещё одна 3D-библиотека
- `swagger-jsdoc` + `swagger-ui-express` — документация API в production bundle
- `recharts` — тяжёлая библиотека графиков
- `sharp` — нативный модуль для обработки изображений (зависимость от платформы)

**Влияние:**
- Раздутый bundle → медленная загрузка в Telegram (особенно в регионах с низкой скоростью)
- Увеличенная поверхность атаки (21 уязвимость в `npm audit`)
- Сложность поддержки и обновлений

**Рекомендация:**
- Разделить `dependencies` на server-only и client-only
- Удалить Three.js/Spline если 3D используется менее чем на 2 страницах — заменить на CSS-анимации или видео
- Перенести Swagger в devDependencies
- Провести tree-shaking аудит — убедиться, что `@radix-ui/*` (17 пакетов!) не попадают целиком в bundle

## 3. СЕРЬЁЗНЫЕ ПРОБЛЕМЫ (P1 — Влияют на рост и качество)

### 3.1 "God Object" в App.tsx (560+ строк)
`App.tsx` выполняет роль роутера, лейаут-менеджера, навигационного компонента, провайдера состояния и обработчика бизнес-логики одновременно. Здесь же определены `NavTab`, `LiquidGlassNav`, `LiquidGlassFilter`, `NonCachedRoute`, `TelegramButtonsSync`.

**Рекомендация:**
- Вынести навигацию в `components/Navigation/`
- Роутинг — в отдельный `Router.tsx`
- Лейаут — в `layouts/MainLayout.tsx`
- App.tsx должен быть <30 строк: провайдеры → лейаут → роутер

### 3.2 Кастомный роутинг вместо стандартного
Приложение использует самописный hash-based роутер (`useRouting.ts`) вместо `wouter` (который уже установлен!). Это создаёт:
- Отсутствие вложенных маршрутов
- Нет route guards
- Нет типизированных параметров маршрутов
- Ручной switch/case в `NonCachedRoute` на 50+ строк

**Рекомендация:** Мигрировать на `wouter` (уже в зависимостях) с типизированными маршрутами.

### 3.3 Дублирование онбординг-систем
Существуют ДВЕ независимые системы онбординга:
1. `OnboardingFlow.tsx` — используется в `App.tsx`, хранит статус в `onboarding_completed`
2. `OnboardingTutorial.tsx` — альтернативная реализация с другим ключом (`onboarding_v1`), A/B-тестированием, и совершенно другой архитектурой

**Влияние:** Пользователь может пройти один онбординг, но увидеть второй. Или наоборот.

**Рекомендация:** Оставить одну систему. Унифицировать ключи хранения и UX-подход.

### 3.4 Отсутствие TON-интеграции
Для Telegram Mini App в 2026 году отсутствие интеграции с TON-экосистемой — серьёзное упущение:
- Нет TON Connect для авторизации через кошелёк
- Нет поддержки TON-платежей (используется только Stripe)
- Нет NFT/SBT для системы достижений/уровней
- Нет on-chain верификации рефералов

**Рекомендация:**
- Внедрить TON Connect 2.0 как альтернативный (или дополнительный) метод идентификации
- Добавить оплату через TON для криптовалютных пользователей
- Рассмотреть SBT для "достижений" в геймификации — это создаст реальную ценность вне приложения

### 3.5 A/B тесты хранятся в памяти
В `server/routes/analytics.ts` результаты A/B тестов хранятся в `Map` (in-memory). При перезапуске сервера все данные теряются.

**Рекомендация:** Перенести в Redis или PostgreSQL.

## 4. ПРОБЛЕМЫ СРЕДНЕГО ПРИОРИТЕТА (P2)

### 4.1 Несогласованный Design System
- Компоненты разбросаны по 4 директориям: `ui/`, `apple-ui/`, `2025/`, `shared/` — без чёткой границы ответственности
- `apple-ui/` и `2025/` перекрываются функционально (оба содержат "премиальные" карточки, кнопки, гласс-эффекты)
- Используется 6+ шрифтов (Cormorant, Inter, Montserrat, Orbitron, Playfair Display, Press Start 2P, Syne) — это чрезмерно для мобильного приложения

**Рекомендация:**
- Консолидировать в единый дизайн-систему с 2-3 шрифтами максимум
- Объединить `apple-ui/` и `2025/` в одну директорию `premium/`

### 4.2 Множество неиспользуемых/устаревших файлов
В корне проекта:
- `QUICK_DEPLOY.md`, `RAILWAY_DEPLOY_FINAL.md`, `RAILWAY_DEPLOYMENT.md`, `RAILWAY_FIX.md`, `DEPLOYMENT_CHECKLIST.md`, `DEPLOYMENT_GUIDE.md` — 6 файлов документации деплоя
- `FULL_CODE_AUDIT_2025-2026.md`, `GITHUB_UPLOAD.md`, `GIT_TEST.md`, `UPLOAD_TO_GITHUB.md` — технический мусор
- `server.js` — дублирует функциональность `server/index.ts`

**Рекомендация:** Удалить все `.md` файлы кроме `README.md` и `replit.md`. Удалить `server.js`.

### 4.3 58 кастомных хуков
58 React-хуков — это признак как зрелости, так и переусложнения. Среди них:
- `useHaptic`, `useHapticManager`, `useScrollHaptic` — 3 хука для вибрации
- `usePerformanceClass`, `usePerformanceMode` — 2 хука для определения производительности
- `usePredictivePrefetch`, `useNavigationTracking` — пересекающаяся функциональность

**Рекомендация:** Консолидировать до 25-30 хуков, объединяя пересекающиеся.

### 4.4 Нет тестов
Ноль юнит-тестов, ноль интеграционных тестов, ноль e2e-тестов. Для продукта, претендующего на мировой уровень, это недопустимо.

**Рекомендация:** Начать с критического пути:
- Telegram Auth flow
- Система рефералов (финансовые последствия при ошибке)
- Gamification/XP расчёты

## 7. ЗАКЛЮЧЕНИЕ

WEB4TG имеет **сильный технический фундамент** — адаптивная производительность, предиктивный префетч, многоуровневая безопасность — это решения уровня enterprise. Однако продукт страдает от "feature creep": попытка объединить витрину, маркетплейс, конструктор, AI-агент, геймификацию, реферальную систему и систему историй в одном Mini App приводит к размыванию ценности и техническому долгу.

**Главная рекомендация:** Определите ONE Thing — одну проблему, которую WEB4TG решает лучше всех в мире. Всё остальное — вторично. Telegram Mini Apps побеждают фокусом, а не количеством фичей.
